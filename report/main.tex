\section{Objective}

This report presents the implementation and comparative analysis of two global path planning algorithms, Dijkstra's algorithm and A*, applied to real-world road networks.
The primary objective is to evaluate and compare the computational performance of the two algorithms in terms of the number of iterations required to find an optimal path, across two cities of significantly different scale: Aosta and Turin.

Dijkstra's algorithm is used as the baseline, as it guarantees an optimal solution on graphs with non-negative edge weights.
A* is then implemented in three variants, each employing a different heuristic function to guide the search:
\begin{itemize}
  \item \textbf{Manhattan distance} 
  \item \textbf{Euclidean distance} 
  \item \textbf{Haversine distance} 
\end{itemize}

For each algorithm and city, ten origin-destination pairs are evaluated and the average number of iterations is recorded.
The results are then discussed in relation to graph size and heuristic choice, with the aim of assessing the practical benefit of informed search over uninformed search.

\section{Design}

\subsection{Graph Construction}

Road networks are obtained from OpenStreetMap using the \texttt{osmnx} library.
For each city, a directed multigraph is constructed, each node represents a road junction and stores geographic coordinates (longitude and latitude), while each directed edge represents a road segment with attributes such as \texttt{length} (in metres) and \texttt{maxspeed} (in km/h).

\subsection{Edge Weight Computation}

The weight assigned to each edge models travel time rather than physical distance, as travel time is a more practical optimization criterion for vehicle routing.
The weight assigned to each edge is:
\[
  w(e) = \frac{\ell(e)}{v_{\max}(e)}
\]
where $\ell(e)$ is the length of edge $e$ in metres and $v_{\max}(e)$ is the posted speed limit converted to m/s.
The resulting weight equals the true travel time $t(e)$ in seconds; the algorithm therefore minimizes total travel time directly.
For edges where no speed limit is recorded in OpenStreetMap, a default value of 40~km/h is assumed.

\subsection{Dijkstra's Algorithm}

Dijkstra's algorithm is implemented using a binary min-heap as the priority queue.
Each node maintains a \texttt{distance} label, initialized to $+\infty$ except for the source, which is set to 0.
At each iteration, the node with the smallest tentative distance is extracted; if it has already been marked as visited, it is skipped; otherwise, all outgoing neighbors are relaxed.
The algorithm terminates as soon as the destination node is extracted from the queue, guaranteeing optimality on graphs with non-negative edge weights.

\subsection{A* Algorithm}
\label{sec:astar_design}

A* extends Dijkstra by incorporating a heuristic function $h(n)$ that estimates the remaining cost from node $n$ to the destination.
The priority of each node in the queue is:
\[
  f(n) = g(n) + h(n)
\]
where $g(n)$ is the exact travel-time cost accumulated from the source to $n$.
Three functions are used to compute the geometric distance $d$:

\begin{itemize}
  \item \textbf{Manhattan}: $d = |x_1 - x_2| + |y_1 - y_2|$

  \item \textbf{Euclidean}:
  \[
    d = \sqrt{(x_1 - x_2)^2 + (y_1 - y_2)^2}
  \]
  The computation is delegated to \texttt{ox.distance.euclidean}.

  \item \textbf{Haversine}: the great-circle distance between two points on the Earth's surface,
  given their latitude $\phi$ and longitude $\lambda$ (the formula is expressed in radians; the implementation via \texttt{ox.distance.great\_circle} accepts decimal degrees and performs the conversion internally):
  \begin{align*}
    a &= \sin^2\!\left(\frac{\Delta\phi}{2}\right)
       + \cos\phi_1\,\cos\phi_2\,\sin^2\!\left(\frac{\Delta\lambda}{2}\right) \\
    c &= 2\,\mathrm{atan2}\!\left(\sqrt{a},\,\sqrt{1-a}\right) \\
    d &= R\,c, \quad R = 6371\text{ km}
  \end{align*}
  The computation is delegated to \texttt{ox.distance.great\_circle}, which operates on geographic (lat/lon) coordinates.
\end{itemize}

To ensure dimensional consistency between $g(n)$ and $h(n)$, each distance-based heuristic estimate is normalized by the maximum speed limit $v_{\max}^{\star}$ observed in the graph:
\[
  h(n) = \frac{d(n,\, \text{dest})}{v_{\max}^{\star}}
\]
For the Euclidean and Haversine variants, this normalization guarantees admissibility: since any road path between two nodes is at least as long as the straight-line distance between them, dividing the straight-line estimate by $v_{\max}^{\star}$ yields a lower bound on the remaining travel time.
The Manhattan variant satisfies $d_{\text{Man}} \geq d_{\text{Euc}}$ and therefore does not carry a formal admissibility guarantee.


All three variants were first evaluated on the original unprojected graph (geographic coordinates in decimal degrees), sharing the same min-heap structure, early-termination condition, and visited-node logic as the Dijkstra implementation.
For the Haversine variant this is correct by design: \texttt{ox.distance.great\_circle} accepts latitude and longitude and always returns a distance in metres, placing $h(n)$ on the same scale as the edge weights.
For the Manhattan and Euclidean variants, however, distances are computed directly from latitude and longitude differences, which lie in the range $10^{-3}$--$10^{-2}$ degrees for typical city-scale paths.
After normalization by $v_{\max}^{\star}$ (in m/s), these values are several orders of magnitude smaller than the accumulated $g(n)$ terms, making $h(n) \approx 0$ and causing A* to explore nodes in the same order as Dijkstra.

To examine how the choice of coordinate system affects heuristic quality, the Manhattan and Euclidean variants were additionally evaluated on a projected version of the graph.
A projected graph is obtained by converting the original geographic coordinates, expressed in decimal degrees of longitude and latitude, to a planar metric coordinate reference system (CRS).
Concretely, \texttt{ox.project\_graph} projects all node coordinates to the appropriate local UTM zone, replacing degree-based values with positions in metres, edge lengths are recalculated in the new coordinate system.
On the projected graph, the $x$ and $y$ coordinate differences used by the Manhattan and Euclidean functions are in metres, placing $h(n)$ on the same scale as the edge weights after normalization by $v_{\max}^{\star}$.
The effect of this coordinate conversion on the number of iterations is evaluated in Section~\ref{sec:results_aosta}.

\section{Experiments}

\subsection{Datasets}

Experiments are conducted on the road networks of two cities: Aosta and Turin.
These cities are selected to represent graphs of very different scales, allowing the comparison of algorithm behavior under both small and large network conditions.
Road networks are downloaded and loaded using \texttt{osmnx} version~2.1.0.
The graph sizes are reported in Table~\ref{tab:graphs}.

\subsection{Experimental Protocol}

For each city, ten origin-destination pairs are randomly sampled from the set of graph nodes.
Dijkstra's algorithm is executed first; any pair for which no path exists is discarded and a new pair is sampled, so that all ten runs correspond to valid paths.
The ten validated pairs are then saved and reused for all three A* variants, ensuring that every algorithm is evaluated on identical inputs and that the results are directly comparable.

\subsection{Evaluation Metrics}

Three metrics are recorded for each run:
\begin{itemize}
  \item \textbf{Number of iterations}: the count of nodes successfully extracted from the priority queue, marked as visited, and whose outgoing neighbors are relaxed, before the destination node is extracted. This is the primary metric for comparing the computational efficiency of the algorithms.
  \item \textbf{Path length} (km): the total length of the reconstructed path in kilometers. This metric is used to verify that the paths returned by A* are optimal, i.e.,\ identical to those returned by Dijkstra on the same pair.
  \item \textbf{Travel time} (s): the total travel time of the reconstructed path in seconds, computed as the sum of the edge weights along the path. Since the edge weight represents the physical travel time, this metric provides a second, independent check of solution optimality: an A* path is confirmed optimal when both its length and its travel time match those of the Dijkstra solution.
\end{itemize}

The average number of iterations over the ten runs is used as the summary statistic for each algorithm–city combination.

\subsection{Discussion}

Path lengths and travel times are identical across all algorithm-run combinations in both cities, confirming optimal solutions on every run.
The sole exception is A* Manhattan (projected) on run~9 in Aosta, which found a suboptimal path.
This is a direct consequence of the Manhattan heuristic not being admissible: by overestimating the remaining cost on run~9, A* discards the true optimal branch before visiting it, committing to a suboptimal solution.
A* Euclidean~(projected), which uses the same projected graph but with an admissible and consistent heuristic, correctly recovers the optimal solution.

The unprojected Manhattan and Euclidean variants produce iteration counts identical to Dijkstra in both cities (0\% improvement).
This confirms the analysis in Section~\ref{sec:astar_design}: coordinate differences in decimal degrees are several orders of magnitude smaller than the edge-weight values, so $h(n) \approx 0$ and A* degenerates to Dijkstra.
Evaluating the same heuristics on the projected graph shows substantially different behavior.
Projected Manhattan achieves the largest reduction: $-59.1\%$ in Aosta and $-43.8\%$ in Turin.
Projected Euclidean and Haversine perform similarly in both cities: in Aosta, $-47.7\%$ and $-47.7\%$ respectively; in Turin, $-35.8\%$  and $-35.7\%$.
The near-identical performance of projected Euclidean and Haversine is expected: at the spatial scale of a single city, the Earth's curvature is negligible, and the great-circle distance computed by the Haversine formula closely approximates the Euclidean distance on a flat plane.

The larger reduction achieved by projected Manhattan compared to projected Euclidean is consistent with the known behavior of overestimating heuristics: by assigning higher $f(n)$ values to nodes farther from the goal, Manhattan focuses the search more aggressively, at the cost of admissibility, as evidenced by the suboptimal result on run~9 in Aosta.

Overall, the results confirm that the effectiveness of A* depends on the dimensional consistency between the heuristic and the edge weights.
When node coordinates are in metres (projected graph), all three heuristic functions provide meaningful guidance; when coordinates are in decimal degrees (unprojected graph), only Haversine produces a usable estimate, since \texttt{ox.distance.great\_circle} always returns distances in metres regardless of the input coordinate system.

The experiments demonstrate that A* can substantially reduce the number of node expansions required to find a shortest path in road networks, regardless of graph scale.
The choice of heuristic plays a crucial role: Manhattan provides the most aggressive pruning, but at the cost of the optimality guarantee; Euclidean and Haversine offer a good trade-off, achieving meaningful reductions while preserving correctness.
